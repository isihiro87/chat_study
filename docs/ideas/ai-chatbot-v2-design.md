> 状態: **アイデア／未実装**（2026-06-23 時点）。実装着手前の構想。
> 目的: 現状の「単発Gemini呼び出し＋静的巨大プロンプト」のチャットボットを、
> RAG・ツール実行・個別化・ガードレールを備えた **層構成パイプライン**へ拡張し、
> 精度・安全・継続率を引き上げる。
>

> - `functions/src/aiChat.ts` … オーケストレーション（レート制限→Gemini→reply→状態書戻し）

## 1. 現状システムと精度の限界

現行（第1世代）: レート制限 → 単一 `gemini-3.1-flash-lite` 呼び出し → reply → 状態書戻し。
全サービス知識を1個の静的プロンプトに内蔵し、`buildSystemPrompt` が学年・教科・直近1問（`users/{uid}.lastQuestion`）を注入。履歴は free 3 / premium 6 ターン。**reply 送信なので LINE 配信枠はゼロ**（コストは Gemini 側のみ）。`DAILY_LIMIT=40`。

精度が頭打ちになる主因:
1. **根拠データを持たない（RAG無し）** → 教科の事実（年号・用語）で幻覚。「私の苦手は？」「範囲設定どうなってる？」に答えられない。
2. **全部を1モデルで処理** → 難問の精度が出ない。
3. **アクションを取れない（tool無し）** → 設定変更・範囲設定・成績取得はAIの外のキーワード機能。
4. **個別文脈が薄い** → 学年・教科・直近1問のみ。弱点・テスト範囲・目標を踏まえられない。
5. **安全策／「配信問題の答えを言わない」がプロンプト頼み** → 確実性が低い。
6. **品質を測る仕組みが無い** → 改善が当てずっぽう。

## 2. 層構成アーキテクチャ（設計案）

| 層 | 役割 | 精度への効き | 費用/複雑さ |
|---|---|---|---|
| ① 意図分類＋ルーティング | 安いモデル/ルールで「配信問題の質問／学校課題／FAQ／設定操作／弱点・成績／雑談・相談／安全要注意／画像音声」に振り分け、専用プロンプト・モデルへ | 中 | 低 |
| **② RAG（カリキュラム接地）★最大** | `data/content/**` と FAQ/運営ドキュメントを埋め込み検索し、教科質問を**正規の解説で根拠付け**して回答 | 大 | 中 |
| ③ ユーザー文脈（個別化） | 事前計算した**ユーザー要約**（学年・testScope・弱点トピック・直近問題・streak・目標）を注入 | 大 | 中 |
| ④ ツール実行（function calling） | `範囲設定` `設定変更` `成績取得` `単元検索` `1問出す` `お問い合わせ引継ぎ` をモデルのツール化 | 中〜大 | 中〜高 |
| ⑤ モデルルーティング/エスカレーション | 雑談=最安、難問や失敗時のみ `gemini-3.5-flash` 等の上位へ | 中 | 低〜中 |
| ⑥ ガードレール/検証 | 入力の安全分類を**決定論的に**安全ハンドラへ。配信問題は lastQuestion 一致を機械判定し**ヒント専用ハンドラに強制**。事実は接地で担保 | 大 | 中 |
| ⑦ 記憶（継続性） | 非同期で**ユーザー要約メモリ**を更新（テスト日・苦戦の継続記憶） | 中 | 中 |
| ⑧ 評価ループ | 会話ログに品質シグナル（聞き返し率・その後の正答・反応）。**実問題のevalセット**で本番前採点 | 大 | 中 |

### スタックへの当てはめ
- **RAGの埋め込みはビルド時に事前計算**して generated ファイル化（既存 `line-scope-index.generated.ts` と同じ思想）。カリキュラムは有限・静的なので Firestore Vector Search でもインメモリ索引でも可。**追加 read ゼロで接地**できる。
- **ユーザー要約は既存の夜間 cron `recalculateUserStatuses` に相乗り**して `users/{uid}.aiContext` に1docだけ書く → 毎ターンのスキャン不要（read規律順守）。
- Gemini ネイティブの **function calling / structured output** でツール層・分類層を実装。
- 多段でも **reply のまま＝配信枠ゼロ**を維持。replyトークンは約1分有効なのでレイテンシ注意（`showLoadingAnimation` 併用、できれば1コールにツール統合）。

## 3. RAG がこのサービスでどう効くか（核心）

RAGの核心は「ボットが**LLMの記憶でなく、自分たちの教材・FAQを根拠に**答える」こと。汎用の賢さでなく「チャットでスタディの教え方と一致した答え」を出せるのが価値。

1. **教材の編集方針とズレない（最重要）**: 例＝「鎌倉幕府の成立に1192/1185の具体年号を断定で入れない（学説併存）」（メモリ `feedback_history_specific_years`）、数学の表記ルール（`feedback_math_quiz_format`）。RAGで該当単元の正規解説を引けば、チャットの答えが**配信問題・暗記カードと同じ事実・流儀**になる。静的プロンプトには全単元の事実を書けないので RAG でしか担保できない。
2. **幻覚（事実誤り）が激減**: 中1〜3 の年号・用語を flash-lite の記憶だけだと外す。教材接地で事故が構造的に減る。中学生は「AIが言うなら正しい」と信じがちで信頼に直結。
3. **配信問題のヒントの質UP（カンニング防止と両立）**: 答えは伏せたまま、単元の暗記カード・解説から**正確な着目点・キーワード・覚え方**を出せる。
4. **問題・単元の「実在」に基づく案内**: 「電池の問題ある？」→ 問題バンクから実在トピック「化学変化と電池」を引き、`questions.topic` と一致した正しい名前で案内。範囲設定・苦手復習と噛み合う。
5. **サービス仕様・FAQをプロンプトから外せる（保守＆コスト）**: 聞かれた仕様だけ引く → プロンプト軽量化・更新はドキュメント修正だけ。
6. **教科が増えても自動スケール**: 新JSONを索引するだけ（`SUBJECT_AVAILABILITY` の coming 教科）。

### RAGが効かない所（正直に）
- 雑談・相談（友だち・部活・進路）… 根拠が無いので不要。
- 設定変更・範囲設定の**実行**… RAGでなくツール実行の領分。
- 数学の**計算そのもの**… RAGは「解き方の型・表記ルール」を接地できるが、計算実行はモデルの推論。

## 4. 会話でのテスト範囲設定（高価値ツールの代表）

KPIで唯一の構造的赤信号が**範囲設定率（新規初日11%）**。会話の流れで設定まで完了できれば最大の弱点に直撃。

フロー: ①意図検知 → ②自然文→**正規トピックに変換**（候補は `line-scope-index.generated.ts` から取得）→ ③確認タップ → ④**既存 `handleScopeCommit` と同じロジック**（era→topic.name 展開・merge write・`testScope.lastSource` 付与）で保存。

### 成否を分ける急所（精度）
**testScope.topics は `questions.topic` と完全一致が絶対条件**（CLAUDE.md 最重要警告）。AIに自由作文で単元名を作らせると存在しない名前になり、「範囲は設定されたのに1問も配信されない」サイレント事故になる。対策:
- モデルには**候補リスト（その学年・教科の実在トピック）を渡し、その中から選ばせる**だけにする（新規文字列生成を禁止）。
- 書き込み前に**索引と照合してバリデーション**。
- **既存 `lineScopeFlow.ts` の純粋ロジックを再利用**（AI経路で範囲書き込みを作り直さない＝`questions.topic` とのズレ再発防止）。`lastSource='ai_chat'` 等を付け、二重送信防止 `onTestScopeSaved`/`onTestScopeFirstSet` に乗せる。
- **対応外（英語＝時代1個・未対応学年）は `/scope` ページや Quick Reply フローへフォールバック**。
- **確認タップ必須**（誤設定＝配信ゼロにつながる）。reply のまま＝配信枠ゼロ。

## 5. できること一覧（★＝費用対効果が高い）

### A. 学習サポートの精度・深さ
- ★教材接地の教科解説（RAG）／★配信問題のヒント高度化
- ステップ式の解き方ガイド（答えは最後本人）
- ★弱点ねらいの類題演習（「問題出して」強化・難易度自動調整）
- つまずき診断（誤答パターンから「時代の前後取り違え」等を指摘）
- 学習プラン提案（テスト日から逆算）／暗記カードの口頭クイズ

### B. 操作を会話でできる（tool use）★全体的に高価値
- ★会話でテスト範囲設定／会話で設定変更（学年・教科・配信時刻）
- 「1問解く」「苦手を復習」「じっくり学ぶ」を会話起動
- ★成績・学習状況の照会（実データ）／月末レポートの会話呼び出し
- 会話で復帰（再開）／★お問い合わせ・人間へのエスカレーション

### C. 個別化・記憶
- ★ユーザー要約メモリ（学年・範囲・弱点・目標・テスト日を継続記憶／cron相乗り）
- テスト日カウントダウン連動／弱点を踏まえた声かけ／口調・テンション適応

### D. モチベーション・継続（KPI: 継続/ブロック）
- ★streak・達成の励まし＆復帰ナッジ（winback連動）
- ランキング/達成の会話共有（手動告知の自動化版）
- つまずき時の感情ケア／週次・月次の自己ふり返り

### E. マルチモーダル（画像・音声の土台あり）
- ★写真から解説（教科書・問題集・ノート。※配信問題以外はOK方針）
- ノートの丸つけ補助／数式・途中式チェック／音声の一問一答（英語リスニング）

### F. 安全・品質・ガバナンス
- ★入力の安全分類（決定論的：自傷・いじめ・個人情報）
- ★カンニング防止の機械判定（lastQuestion一致でヒント専用へ）
- 事実グラウンディング（年号・用語を教材で裏取り）
- ★要注意会話の自動フラグ→運営（夫婦）に通知／保護者連携（慎重運用）

### G. 運営・改善ループ
- ★会話品質の計測（聞き返し率・その後の正答・反応）
- evalセットで本番前採点／答えられない質問→教材の穴を運営に提案
- FAQ自動拡充／A/Bテスト基盤

### H. 収益・拡張（マネタイズ議論と接続）

## 7. 守るべき制約・規律（実装時の必須前提）
- **配信枠ゼロを維持**: reply 送信を貫く（push は使わない）。コストは Gemini 側のみ。
- **Firestore read 規律**: 候補リスト・RAG索引はビルド時生成（追加readゼロ）。ユーザー文脈は cron 事前計算の1docを読む。毎ターンの answers/users スキャン禁止（CLAUDE.md §読み取りコスト規律）。
- **testScope は `questions.topic` と完全一致**（§4）。範囲書き込みは既存 `lineScopeFlow.ts` を再利用。
- **2人運用**: 全部一気に作らない。第一弾を絞って段階導入。
- **公式LINEの単元は `data/content/` が正**（`eras`/`topic-registry` は Web版専用・混同禁止＝CLAUDE.md冒頭）。
