## 前提となる現状認識

数字はいずれも良好（オンボ完了93.8%、D7継続54.9%、週1学習UU比率65%、
登録1,929・週1学習UU 1,260・ブロック率11.1% @2026-07-01）。
ただし**構造的な問題が2つ**あり、これが優先順位を決める。

1. **配信枠クライシス**: 月30,000通の上限に、基準シナリオで**7月31日に到達**（8月は109%超過）。
   オンボ期の毎日pushだけで月18,600通＝枠の62%が「動かない床」。
2. **収益ゼロで原資がない**: 過去の有料化は trial→課金転換0件・申込1件で撤退済み。
   枠の恒久解「上位LINEプラン購入」には収益化が前提。

→ 「改善点」と「有料化」は独立ではなく、**有料化が配信枠問題の恒久解の原資**という関係。

## 改善点（優先順）

### P0: 今月中に着手すべき（構造的リスク・実測データの裏付けあり）

**1. コア学習ループの reply 化（配信枠対策の本丸）**
毎日配信 push の一部を「ユーザー起点の pull」に置き換える。
- オンボ期の毎日push（7日間）を5日に短縮、または3日目以降は「回答した人だけ翌日も届く」条件付きに
- 回答後の「つづけて学ぼう」flex に**「明日の分を今すぐ解く」**を追加
  （reply なので枠ゼロ。追加学習率19.1%の層は push を待たず消化してくれる）
- リッチメニューに「今日の1問」ボタン常設（push 未達・既読スルー層の pull 導線）

**2. 範囲設定をオンボーディングの正式ステップに組み込む**
✅ **調査の結果ほぼ実装済みと判明（2026-07-02）**: オンボ完了 reply に scope ガイド＋時代チップを
同梱する実装が既にあり、英語も学年別再構成（`3c911512`）で era フロー対応済み。
残っていた「単元0 の教科×学年」向けフォールバック導線のみ今回追加した。
新規初日の範囲設定率11%が全KPIの最大の穴。設定群は未設定群と比べ
**ブロック率半減（8.8% vs 18.1%）・D7継続+32pt・回答数2.8倍**。
現在は「オンボ完了後の任意ボタン」→ 学年→教科→時刻→**範囲（既存 scope_start の
トーク内 Quick Reply フローを流用）**→完了、と第4ステップに昇格させる。
「あとで設定」の逃げ道は残しつつデフォルトを設定側に。実装コストは低い（呼び出し順変更が主）。

**3. 友だち追加直後に「その場で1問」**
ブロックの46%が登録48時間以内、無回答層のブロックは28.7%。
ウェルカムメッセージに学年不問のサンプル1問を同梱し、初回配信を待たずに成功体験を作る。
回答 postback は reply なので枠ゼロ。「1問も解かずに切る」層への唯一の接触機会。

**4. D30継続率と開封率の計測開始**
登録の82%が直近30日に集中＝**急成長の定着はまだ未検証**のまま規模だけ拡大している。
7月から登録30日超コホートが出るので D30 を KPI ダッシュボードに追加。
開封率は OA Manager から週次で手動転記でよいので開始（自認済みの「計測の穴」2点）。

### P1: 数週間以内（明確な機会損失の解消）

**5. 配信停止・再開のユーザー選択（2026-07-02 追加・ユーザー指示）** ✅**実装済み（2026-07-02、デプロイ待ち。`.steering/20260702-line-delivery-pause-and-pull/`）**
毎日/週3配信をユーザー自身がオン・オフできるようにする。効果は三重:
① **ブロックの受け皿** — 現在「配信を止めたい」の唯一の手段がブロック（11.1%、無回答層は28.7%）。
「一時停止」という逃げ道があれば友だち関係を維持したまま休める（テスト後・長期休み等）。
② **配信枠の節約** — 停止中ユーザーは push ゼロ＝枠クライシス対策と同方向。
③ **Win-back との連携** — Day14 最終メッセージ等に「配信を一時停止する」選択肢を添えれば、
churned 層をブロックさせずに静観プールへ移せる。
実装は軽い: 設定・サポート flex ＋リッチメニューに postback `type=pause_delivery` →
`users/{uid}.deliveryPaused=true` を立て、dailyQuiz cron / win-back の対象クエリで除外。
再開は既存の復帰キーワード「再開」（`detectRestartIntent` 実装済み）とリッチメニューの
postback を流用し、再開時は即1問 reply。停止時に「いつでも『再開』と送ってね」を必ず案内する。

**6. 英語の範囲設定をトーク内フロー対応に** — 英語だけ `/scope` 外部ブラウザ＋OAuth へ
フォールバック＝最も離脱しやすいUX。正答率61.8%で利用実態はあるだけに惜しい。
単元メタを2階層に整理して scope_start フローに乗せる。

**7. 月末ふり返りレポート cron の本番投入** — 本体は完成済み・未デプロイ
（`sendMonthlyReportInvite`）。月1回の push 消費（当月回答者のみ）はあるが、
「自分の頑張りが可視化される」体験は win-back より継続に効く見込み。
まず7月末に投入して開封反応を計測（アーム前にユーザー承認要 = CLAUDE.md 記載）。

**8. テストモード（テスト日程登録）** — 「テスト日」を Quick Reply で登録→
テスト2週間前から出題を範囲集中・頻度アップ、前日に総まとめ、終了後に通常へ復帰。
中学生の学習動機は定期テスト駆動（ペルソナ定義とも一致）なのに、
現在の設計は「毎日一定」でテスト山に同期していない。AIスケジュール（P2-11）の土台。

**9. Win-back の効果測定とスリム化** — Day3/5/7/10/14 の5接点は設計過多気味（枠の6.5%消費）。
受信者の28%が再回答という実績はあるので、touchpoint 別の復帰率を出して効かない回を落とす
（Day3 停止は時間稼ぎ策としてダッシュボードにも記載あり）。

### P2: 体験の深化（1〜2ヶ月）

**10. ムビスタ連動: 誤答→授業動画への導線（連動構想の第一歩）**
誤答時の解説 reply に該当単元のムビスタ授業動画リンク（`u/<topicId>.html`——
**共有ページ・OGP画像は生成済み**）を添える。reply 内リンクなので枠ゼロ。
「1問→間違えた→3分授業で理解→もう1問」のループは追加学習率を直接押し上げる。
ムビスタ側の視聴開始/完了を Firestore に記録すれば AIスケジュールの入力になる。
marutto 側の `/mubista` プロキシは構築済みで UI 導線が未実装なので、まずここから。

**11. AI学習スケジュール（ムビスタ×AIチャットの統合機能）**
テスト日＋範囲＋弱点（weakReview）＋ムビスタ視聴履歴から Gemini が週間プランを生成:
「今日は『明治維新』の動画3分＋クイズ5問」。毎日配信を「無差別1問」から「今日のメニュー」へ。
実装は AIチャットv2 構想（`docs/ideas/ai-chatbot-v2-design.md`）のツール実行・ユーザー文脈の応用で、
月末レポートのパターン（集計→Gemini→reply、キャッシュで課金抑制）を流用可能。
**有料化の目玉機能でもある（後述）。**

**12. ゲーミフィケーションの拡張** — 現状は streak とマイルストーンのみ。費用対効果順に:
① streak 救済（週1回まで欠席免除——リセットの喪失感はブロック誘因）
② 週間目標（「今週20問」進捗バー）
③ 称号（正答率×単元で「江戸マスター」等、成績 flex に表示）
④ 学年別匿名ランキング（opt-in）
すべて reply で完結し枠ゼロ。

**13. AIチャットv2（RAG・会話でのツール実行）** — 特に「会話で範囲設定・設定変更」の
ツール実行は範囲設定率にも効く二重の価値。構想ドキュメント化済み・未着手。

**14. 保護者連携（有料化の前提部品）** — 親の LINE を連携コードで紐付け、
子の週間サマリー（回答数・正答率・streak）を送る無料機能をまず作る。
過去のプレミアム申込が UU2 で止まった主因は「**支払者である親に情報が届いていなかった**」こと。
有料化の成否はこの接点の有無で決まる。

**15. 教科の複数登録** — 現在1ユーザー1教科固定。5教科あるのに1教科しか届かないのは
定期テスト（5教科同時）の実態と乖離。配信は主教科のみ・追加で解く/範囲設定は
複数教科対応、なら枠を増やさず実現可能。

### P3: 技術負債・運用（随時）

16. trial 残置コードの削除（trialDrip 群・expireTrialUsers 等。誤動作リスクと discovery timeout の一因）
17. Functions v2 個別 import 化 / codebase 分割（デプロイ timeout の根本解決）
18. Instagram Meta シークレット登録（全体デプロイが常に失敗する状態の解消）
19. リッチメニューの free/premium 二重管理の整理（無料化済みなのに2系統が残る）

### 保留（2026-07-02 ユーザー判断）

- **数学の死蔵解消** — 直近7日で回答1問のみ＝1教科がほぼ機能していない。
  公開時期を決めるか、当面出せないなら教科選択肢から外す（選んで問題が来ない体験が最悪）。
  → **当面保留。**再開判断のトリガー: 数学コンテンツ生成の完了、または数学選択ユーザーの苦情/離脱が観測されたとき。

## 有料化案

### 大原則（過去の実績から）

- **中学生本人は払わない**: trial→課金転換0件が実証済み。価格ロック・希少性演出などの
  セールス設計は良質だったのに0件＝手法でなく**支払者の選定**が誤り。
- **支払者は保護者**。「親に届く接点」（P2-14）が先行投資として必須。
- **「子どもの学習は無料のまま」は死守**: 巻き戻しはブロック急増と信頼毀損を招く。
  売るのは「親向けの追加価値」と「上位体験」。

### 案A（本命）: 保護者プラン ¥980/月

子どもは全機能無料のまま、親に売る。特典:
- **週次AI学習レポート**（回答数・正答率推移・苦手単元を Gemini 分析、親向け文体で）
- **AIスケジュールの共有**（「今週のテスト対策プラン」を親にも配信 → P2-11 と直結）
- **テスト前アラート**（「テスト10日前です。理科の◯◯が弱点です」）
- **ムビスタ視聴レポート**（どの授業を何分見たか → 連動構想が課金価値に変わる）

原価ほぼゼロ（Gemini 集計＋reply/narrowcast）。塾の月謝1〜3万円との対比で¥980は
説明しやすく、「三者面談が毎週LINEで届く」訴求。
**週1学習UU 1,260人の親の3%転換で月¥37,000**＝LINE上位プラン（枠拡張）の原資として成立。

### 案B（即効・検証を兼ねる）: クラファン先行販売の実施

`docs/ideas/crowdfunding-parent-plan-presale.md` に**設計完了済み・未実施**。
¥3,980（6ヶ月）/¥6,800（12ヶ月＋価格ロック）の保護者プラン先行販売が本命 tier。
配信枠が逼迫する7月中に公開すれば ①原資 ②「親は払うか」の実証 ③PR を同時に取れる。
トラクション数値は当時の設計（登録1,177）から**登録1,929・累計回答49,654問**に更新して訴求力up。
**案Aを本実装する前の需要検証として、順序は B→A を推奨。**

### 案C: プレミアム学習体験（子ども向け・親決済）

無料の骨格は変えず、上位体験を親決済で:
- **AIパーソナルスケジュール**（毎朝「今日のメニュー」: 動画＋問題の個別プラン）
- **ムビスタ応用トラック**（応用ナレーション・応用スライドは別トラック実装済み→有料枠にしやすい）
- **記述問題のAI採点**（ムビスタ server モードの Claude 採点。**原価がかかる機能＝課金と最も相性が良い**）
- AIチャット優先枠（1日40回制限の緩和）

案Aとバンドルして「ファミリープラン ¥1,480」も可（クラファン¥30,000 tier の多子世帯需要と整合）。

### 案D: B2B（塾・学校ライセンス）

クラファンの¥100,000 tier をリード獲得の入口に。塾向けには
「生徒一括登録＋講師ダッシュボード（生徒別の回答状況）」で月額課金。
運営が塾経営という強み・人脈がそのまま営業チャネル。
単価が大きく配信枠原資として最も効率的だが、管理画面開発が必要なので中期。

### 非推奨

- **広告・アフィリエイト**: 中学生向け教育サービスでは信頼毀損リスクが単価に見合わない。
- **子ども本人向けサブスク復活**: 実証済みの失敗。親決済の文脈なしには再挑戦しない。

## 位置づけメモ

ムビスタ×AIスケジュール×LINE連動は、**単なる機能追加ではなく「有料化の目玉」に
位置づけるのが最も効果的**（無料側は「誤答→動画リンク」で価値を見せ、
パーソナルスケジュール化から先を保護者プランに載せる）。

次に着手する候補: 実装コスト最小で効果実証済みの「範囲設定のオンボ組込（P0-2）」、
または連動の第一歩「誤答→ムビスタ動画リンク（P2-10）」。
